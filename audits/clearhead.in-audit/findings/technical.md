# Technical SEO Audit — clearhead.in

**Date:** 2026-07-14
**Scope:** robots.txt, sitemap.xml, canonicals, redirects, security headers, URL structure, mobile config, crawlability, JS-rendering dependency, and indexability across homepage, pricing, blog index, 24 blog posts, 5 tool pages, and legal pages.

**Method note:** This session's tool access did not include a live HTTP fetch/render capability, so this pass was done entirely against the local source mirror at `/Users/jain/Documents/Zen`, which per the brief is what's actually deployed to Netlify (build config: `publish = "."`, no build step). Findings on markup, meta tags, structured data, robots.txt, sitemap.xml, and `netlify.toml` (which governs live response headers and redirects) are drawn directly from the files Netlify serves. Anything that can only be confirmed by inspecting actual live response headers (e.g., whether Netlify's default TLS/HSTS behavior differs from what's configured) is flagged below as unverified rather than asserted as pass/fail. Per the brief, the 3 unpushed blog posts / sitemap updates are a known, already-flagged issue and are not re-reported here.

## Summary

The site is in good technical shape overall: clean flat URL structure, unique and well-formed titles/descriptions on all 40 pages checked, consistent canonical tags, correct `noindex` on legal/utility pages, valid JSON-LD (Person/LocalBusiness/FAQPage/Article/SoftwareApplication/BreadcrumbList) on the pages that need it, a fresh and complete sitemap, and no client-side-rendering dependency for primary content anywhere on the site — this is a static HTML site in the truest sense, so Googlebot (and any other crawler) sees full content on first fetch. The one previously-known issue (robots.txt `Disallow` conflicting with `noindex` on `/thank-you.html` and `/quiz.html`) is confirmed still open. The new findings from this pass are mostly around security-header completeness and one real Core Web Vitals risk: blog post hero images are pulled from `images.unsplash.com` with `loading="eager"` (i.e., they're the LCP candidate) but the site never preconnects to that origin, so every post pays full DNS+TCP+TLS setup cost on the largest paint element.

## Findings

| Issue | Severity | Evidence | Recommendation |
|---|---|---|---|
| robots.txt `Disallow` conflicts with `noindex` meta on the same two URLs (known, already open) | Low | `robots.txt:3-4` disallows `/thank-you.html` and `/quiz.html`; both files carry `<meta name="robots" content="noindex, nofollow" />` (`thank-you.html:9`, `quiz.html:9`). Because the pages are disallowed, Googlebot can never crawl them to see the `noindex` directive — if either URL ever gets an external link, Google can still index the bare URL with no snippet ("indexed, though blocked by robots.txt") instead of cleanly dropping it. | Pick one mechanism, not both: remove the two `Disallow` lines and let `noindex` do the deindexing (preferred, since it actively deindexes rather than merely hiding from crawl), or drop the `noindex` tags and rely solely on `Disallow`. |
| No `Strict-Transport-Security` header configured | Low | `netlify.toml:36-43` sets `X-Content-Type-Options`, `X-Frame-Options`, `Referrer-Policy`, `Permissions-Policy` — no HSTS entry. Netlify enforces HTTPS by redirect but does not add HSTS by default unless it's explicitly set. | Add `Strict-Transport-Security = "max-age=31536000; includeSubDomains; preload"` to the `[headers]` block in `netlify.toml`, then submit the domain to the HSTS preload list once confirmed stable. |
| No `Content-Security-Policy` header, on a domain that runs Razorpay checkout | Low-Medium | `netlify.toml:36-43` has no CSP directive. `pricing.html:193` loads `https://checkout.razorpay.com/v1/checkout.js` (a payment-flow script), and pages also pull Google Fonts and `images.unsplash.com`. Without a CSP, there's no defense-in-depth against injected/malicious script on the one page that handles payment initiation. | Add a CSP header (start in `Content-Security-Policy-Report-Only` mode to avoid breaking the fonts/Unsplash/Razorpay/Cal.com integrations, then enforce). At minimum allow-list `checkout.razorpay.com`, `fonts.googleapis.com`, `fonts.gstatic.com`, `images.unsplash.com`, and Netlify Forms' own origin. |
| Blog post LCP image is loaded from a third-party CDN with no `preconnect`, adding avoidable latency to the largest paint element | Medium | All 24 blog posts follow the same pattern seen in `post-perfectionism.html:130`: `<img src="https://images.unsplash.com/photo-...&w=1400..." loading="eager" ...>` as the hero/LCP image, while the `<head>` only preconnects to `fonts.googleapis.com` and `fonts.gstatic.com` (`post-perfectionism.html:32-33`) — no `<link rel="preconnect" href="https://images.unsplash.com">`. The homepage isn't affected (its hero uses local `VJ.jpg`, `index.html:229`). | Add `<link rel="preconnect" href="https://images.unsplash.com" crossorigin>` to the `<head>` of the post template used for all `post-*.html` files. Longer-term, consider self-hosting/optimizing the hero images (WebP/AVIF, Netlify Image CDN, or downloading + committing them) to remove the third-party dependency entirely and get tighter control over LCP. |
| `/index.html` and `/` are both servable with no explicit redirect consolidating them | Low | `netlify.toml`'s `[[redirects]]` rules only cover `www→non-www` and the `/api/*` function proxies — there's no rule redirecting `/index.html` to `/`. The `rel=canonical` on `index.html:9` correctly points to `https://clearhead.in/`, which mitigates most risk, but Netlify does not automatically fold `/index.html` into `/` for a static `publish = "."` site, so both URLs remain independently crawlable. | Add a redirect rule `from = "/index.html"`, `to = "/"`, `status = 301`, `force = true` to `netlify.toml` for defense-in-depth beyond the canonical tag. |
| Nav "Book a free call" pill (`.cta-ghost`) is under the ~44×44px touch-target recommendation on mobile | Low | `styles.css:73-76` sets `.cta-ghost { padding: 0.4rem 0.8rem; }`; the `@media (max-width: 560px)` rule at `styles.css:77-79` further shrinks nav link font to `0.88rem` and only hides the *first* nav link, leaving the ghost CTA at reduced padding/font on small screens — effective tap height is meaningfully under 44px. | Bump `.cta-ghost` padding (e.g., `0.65rem 1rem`) at the `max-width: 560px` breakpoint specifically, or apply `min-height: 44px; display:inline-flex; align-items:center;` so the tap target meets guidance without changing the visual density elsewhere. |
| IndexNow (Bing/Yandex/Naver) not implemented | Info | No IndexNow key file (`/*.txt` key at root) or ping call found anywhere in the repo or Netlify Functions. Sitemap shows an active weekly posting cadence (3 posts added in the last week per `sitemap.xml` lastmod dates). | Given the blog's publishing velocity, add a lightweight IndexNow ping (a small Netlify Function or a step in the deploy hook) that notifies `api.indexnow.org` on new/updated URLs — this is a low-effort win for faster Bing/Yandex pickup that doesn't require any new infrastructure since Functions are already in use. |
| Tool-page personalized results are not present in static HTML (input-gated by JS) | Info | `runway.html` and the other 4 calculators (`bandwidth.html`, `career-friction.html`, `ai-relevance.html`, `readiness.html`) render intro copy, questions, and page shell statically, but the computed output (e.g., `runway.html:1014-1026` — the result messages) only exists inside JS string literals and is populated at runtime after user input. This is expected/acceptable UX for an interactive calculator and not a defect, but it means the "result" content itself is not indexable or visible to a no-JS crawler — only the surrounding educational copy is. | No action required for correctness. If there's ever a desire for the result *content itself* to be indexable/shareable (e.g., "your runway is X months" as a landing page), consider a shareable results URL with query-string or slug-based state that pre-renders a summary server-side — optional, not a current problem. |
| Homepage/footer email address requires JS to render at all | Info | `index.html:571` replaces `.js-email` spans with a `mailto:` link only via inline script after page load; if JS fails to execute, the visible email address is simply absent (empty `<span>`). This is intentional spam obfuscation, not a rendering-dependency problem for Googlebot (which executes JS), but worth noting as the one piece of contact info that isn't in the raw HTML. | Low priority — acceptable trade-off for spam protection given WhatsApp and the contact form are both present in raw HTML as alternate contact paths. No change needed unless no-JS accessibility becomes a stated requirement. |

## What's already strong

- **Sitemap is clean and fresh**: 32 URLs, all indexable pages present, `lastmod` dates current (up to 2026-07-12), correctly excludes `thank-you.html`, `quiz.html`, `privacy-policy.html`, and `terms.html`.
- **No orphaned or duplicate `noindex`/canonical conflicts** outside the one known thank-you/quiz case — legal pages (`privacy-policy.html`, `terms.html`) are correctly `noindex, follow` *and* crawlable (not blocked in robots.txt), so Google can actually see and honor the noindex.
- **Zero JS-rendering dependency for primary content** anywhere on the site — every page's core text, headings, and internal links are present in the raw HTML source. This is a genuinely rare "pass" for a modern site.
- **Titles and meta descriptions are unique across all 40 pages checked**, appropriately length-constrained, and written distinctly per post (no templated duplication) — good defense against the "thin/duplicate content" failure mode common on programmatically-generated blogs.
- **Structured data is well-formed and appropriately scoped**: `Person`/`LocalBusiness`/`ProfessionalService`/`FAQPage` on the homepage, `Article` with citations on posts, `SoftwareApplication` + `BreadcrumbList` on tool pages, `CollectionPage` + `BreadcrumbList` on blog/tools index.
- **Images have explicit `width`/`height`** attributes and correct `loading="eager"` vs `loading="lazy"` split (hero eager, below-fold lazy) throughout, which is the right pattern for CLS prevention.
- **robots.txt doesn't block any AI crawlers** (GPTBot, Google-Extended, PerplexityBot, ClaudeBot, etc.) — the wildcard `Allow: /` covers them by default, which is the right call for a coaching practice that wants AI-assistant visibility.
- **Redirect setup is minimal and clean**: single-hop `www → non-www` 301, no chains found in the config.
- **Security headers baseline is present**: `X-Content-Type-Options`, `X-Frame-Options: DENY`, `Referrer-Policy: strict-origin-when-cross-origin`, and a reasonable `Permissions-Policy` are all set in `netlify.toml`.
